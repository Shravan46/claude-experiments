Section 01

Feature Overview

An AI assistant embedded in DXM Vision's Manufacturing Execution System (MES) that proactively identifies production orders at risk of missing their due dates — before the delay becomes irreversible.

Manufacturing floors generate continuous signals: machine faults, inventory alerts, shift callouts. Today these signals live in separate dashboards. A CNC spindle fault fires in the machine monitoring system. A component stockout fires in inventory management. A welder callout registers in the workforce schedule. None of these systems ask the next question: which active production orders does this affect, and how badly?

The result is reactive operations. Supervisors find out an order is in jeopardy when it is already late — not when there was still time to reroute, re-staff, or expedite. The AI assistant closes this gap by synthesizing signals across three domains — machine health, inventory availability, and workforce — and attributing risk at the order level with a severity rating and a one-sentence rationale.

Who It Serves

Persona Primary Need How the Feature Helps
Production Planner Early, prioritized risk signals to protect schedule Receives a ranked list of at-risk orders with root-cause attribution before shift handoff
MES Supervisor Actionable flags tied to specific orders, not system alerts Every flag is order-centric: "PO-4821 is at risk because CNC-07 is down" rather than "CNC-07 is down"
Operations Manager Visibility into cascade risk and recovery options Secondary driver identification surfaces compounding risks before they become primary constraints

Why It Exists

DXM Vision's current monitoring posture is reactive. The gap is not data availability — real-time telemetry, inventory feeds, and workforce schedules already exist. The gap is synthesis: no system currently translates isolated operational alerts into order-level risk with enough lead time to act. This feature is the synthesis layer.

Section 02

Eval Objectives

This eval suite is designed to answer five questions before we authorize production deployment.

# Question Tested By
Q1 Does the model correctly identify at-risk orders when signals are clear and unambiguous? Happy Path (HP-001 – HP-005)
Q2 Does the model calibrate severity accurately — distinguishing a critical order miss from a recoverable risk with buffer? Edge Cases (EC-001 – EC-004)
Q3 Does the model hallucinate risk from alarming-but-irrelevant signals that do not apply to active orders? Adversarial (AV-001 – AV-003)
Q4 Does the model catch secondary risk drivers — the quieter compounding factors alongside the obvious loud signal? Regression (RG-001 – RG-003)
Q5 Is the model safe to put in front of production planners — meaning its worst failure mode is missing a risk, not inventing one? Adversarial + overall critical failure count
Design Principle
In a manufacturing context, a false positive (flagging a safe order as at-risk) causes wasted supervisor attention. A false negative (missing a genuinely at-risk order) causes a late shipment. A hallucinated risk on an order that is actually safe and well-buffered is the worst outcome: it erodes trust in the system, causing operators to dismiss future alerts — including legitimate ones. Q5 is therefore the gating question. An unacceptable adversarial failure rate blocks launch regardless of performance on other categories.
Section 03

Test Case Design Rationale

Four categories. Each tests a distinct failure mode. Together they cover the full risk surface of a production deployment.

Happy Path (HP-001 – HP-005, 5 cases)
Baseline Competence

Clear, unambiguous signals where a correct answer requires no inference beyond the data provided. Machine is down and the order runs on that machine. Inventory will run out before the order completes. Certified operators are absent and the operation requires certification. If the model cannot achieve near-perfect scores here, no further evaluation is warranted.

HP-004 adds one layer of complexity: three orders in the snapshot, two at risk and one on track. This tests whether the model correctly applies the "no risk inferred beyond data present" constraint — it must flag the right two orders and leave the third alone.

HP-005 tests OEE-to-throughput reasoning. The machine is not down; it is degraded. The model must perform the math (57% OEE × 2.0 hr standard = ~3.5 actual hr) and compare it to the remaining time window. This is the most cognitively demanding of the happy path cases.

Edge Cases (EC-001 – EC-004, 4 cases)
Severity Calibration & Multi-Driver Detection

Cases where the correct answer requires judgment rather than pattern-matching. EC-001 presents two degraded-but-not-catastrophic signals and expects the model to name both as contributing drivers with medium severity. EC-002 presents a downed machine but gives the order ~12 hours of float — the correct severity is low, not high. EC-003 uses a component that is nominally sufficient (201 vs. 200 required) but historically risky given a 2.1% reject rate. EC-004 introduces cascade risk: solving the primary order's workforce gap by reallocating operators would create a new risk for a second order.

This category is designed to surface severity over-escalation bias — the tendency to call everything "high" when any risk signal is present — and driver incompleteness — catching the primary signal but missing the secondary one.

Adversarial (AV-001 – AV-003, 3 cases)
Hallucination & Relevance Filtering

Each adversarial case includes an alarming signal that is genuinely irrelevant to the active orders in the snapshot. AV-001: a machine exhibiting anomalous vibration and temperature, but no active order is running on that machine. AV-002: a critical component inventory alert, but the component is exclusively used in a product line with no active orders. AV-003: a workforce shortage alert for a certified operation that has already been completed on the order in question.

The correct answer for all three is: no orders currently at risk. The model is permitted to note the signals as future planning concerns. It is not permitted to flag active orders as at-risk based on inapplicable data. Any hallucinated risk flag is scored as 0.0 with no partial credit.

This category is zero-tolerance because in production use, a hallucinated flag from alarming-but-irrelevant data is exactly how operators learn to ignore the system. One high-profile false alarm on a time-pressured shop floor destroys weeks of trust-building.

Regression (RG-001 – RG-003, 3 cases)
Secondary Driver Detection

Each regression case contains one obvious, "loud" risk signal and one quieter, secondary signal that a shallow analysis would miss. RG-001: an obvious primary victim of a downed machine, plus a second queued order that depends on the same machine and is often overlooked. RG-002: a severe inventory shortfall (the loud signal) plus a workforce certification gap on the next operation that would prevent recovery even if the inventory problem were resolved. RG-003: an obvious workforce staffing gap (the loud signal) plus a declining OEE on the assigned machine that compounds the risk independently.

Missing a secondary driver in a regression case is scored as partial credit (0.5). This reflects the real operational cost: a planner who receives an incomplete risk picture may resolve the primary constraint and discover the secondary one too late to act.

Section 04

Scoring Rubric

Three score values. Each reflects a distinct quality of response. The thresholds are strict because the cost of ambiguity in production is a late shipment.

Score Verdict General Criteria Category Exceptions
1.0 Pass Correct order identification, correct primary driver, correct severity, reasoning grounded in snapshot data only. Adversarial: correctly clears all active orders; may note signals as future concerns. Regression: both primary and secondary drivers named and attributed to the right order.
0.5 Partial Correct order identification and primary driver, but one of: wrong severity, missing secondary driver, or minor diagnostic gap in rationale. Adversarial: no partial credit — hallucinating risk for a safe order is 0.0. Regression: missing the secondary driver caps at 0.5 regardless of how well the primary is handled.
0.0 Fail Misses an at-risk order entirely, misidentifies the primary driver, or flags an order that is not at risk. Adversarial: any hallucinated risk on an active order that is actually safe = 0.0, no exceptions. Happy Path: failure to identify the at-risk order = 0.0.

Rationale for Three-Tier Scoring

A binary pass/fail rubric would mask an important distinction: a model that identifies the right order with the wrong severity is significantly more useful than one that misses the order entirely. The 0.5 score preserves this signal, allowing the eval summary to differentiate a model that has the right instinct but poor calibration from one that is making fundamental attribution errors.

The adversarial exception to partial credit is intentional and non-negotiable. Severity over-escalation on a legitimate at-risk order is a calibration problem, fixable with prompt tuning. Hallucinating risk on a safe order is a relevance-filtering failure — a categorically different problem that cannot be partially rewarded.

Section 06

Critical Failure Modes

Three failure types are classified as zero-tolerance. Any single instance blocks launch, regardless of overall score.

Failure Mode Definition Why Zero-Tolerance Run 1 Instances
Cross-Machine Hallucination Model flags an active order as at-risk because of machine anomalies on a machine not assigned to that order. Planners will act on this — rerouting an order that is actually on schedule burns capacity and creates disruption where none existed. 1 (AV-001)
Completed-Operation Hallucination Model flags an order as at-risk for a risk driver that applies only to an operation that has already been completed on that order. This is the hardest hallucination to defend against with UI guardrails — the alert looks credible, the order is real, and the operation name is correct. Only the completion status distinguishes it. 1 (AV-003)
False Flag on Confirmed Safe Order Model assigns at-risk status to an order with no active risk drivers, fabricating a risk scenario from data that does not apply to that order. Once planners see the system produce a false alarm on an order they know is safe, every subsequent alert — including legitimate ones — is viewed with skepticism. Trust is asymmetric: slow to build, fast to lose. 1 (AV-001, PO-3388)
Non-Zero-Tolerance Failures (Still Important)
Severity over-escalation (assigning HIGH to a low-buffer situation) and driver ordering errors (inverting primary and secondary) are not zero-tolerance, but they are tracked as quality metrics and have their own remediation actions. If either pattern appears in more than 30% of applicable cases in Run 2, the threshold should be re-evaluated before the subsequent run.

Section 08

Root Cause Analysis

Run 1 reveals two distinct failure patterns. They have different root causes and require different fixes. Conflating them will produce a prompt change that partially addresses one and worsens the other.

Pattern 1: Compulsion to Flag When Alerts Are Present

Both adversarial failures share a common mechanism: the model encountered a genuinely alarming signal in the snapshot, correctly reasoned that it was irrelevant to active orders — and then flagged something anyway.

In AV-001, the model correctly identified CNC-12's telemetry anomalies as not affecting current orders. It then fabricated a different machine risk (LATHE-01 schedule tightness) to justify flagging PO-3388 — a risk that was not present in the data. In AV-003, the model correctly explained in its rationale that the Shift 3 coating alert was inapplicable to PO-4455 because Op 30 was already complete. It then entered "AT RISK — LOW severity" in the structured output field regardless.

This suggests the model has an implicit prior that a snapshot containing alert-level signals should produce at least one at-risk flag. When it cannot find a legitimate target, it manufactures one. This is not a reasoning failure — the model's reasoning in both cases was largely correct. It is a conclusion-anchoring failure: the structured output format creates pressure to produce a risk flag, and the model satisfies that pressure even when its own analysis contradicts it.

Pattern 2: Severity Defaults to HIGH Under Any Alert Condition

Three of four edge case partial-credit results share the same failure mode: the model assigned HIGH severity when the correct answer was MEDIUM or LOW. This occurred on EC-001 (two degraded signals with recovery options), EC-002 (machine down with ~12 hours of float), and EC-004 (workforce gap with a reallocation option).

The model's severity logic appears to be: if a risk driver is present and the order will require intervention, severity is HIGH. It does not discount severity for available buffer, viable recovery paths, or the difference between "intervention required" and "intervention required immediately or the order is already lost." This is a severity calibration failure — the model is using a binary risk-present/risk-absent frame where the rubric requires a three-tier frame (high/medium/low) based on time pressure and recovery optionality.

Pattern 3: Machine Signal Dominance in Multi-Driver Cases

RG-003 reveals a subtler pattern: when both machine and workforce signals are present, the model weights machine OEE degradation as the primary driver even when the workforce gap is the more constraining factor. This may reflect training data patterns in which machine signals are more commonly the primary driver, or a structural bias toward the first well-quantified signal encountered in the input. This pattern needs one more data point before it is classified as systematic.

What the Pattern Does Not Show

The model's reasoning on legitimate risk cases is strong. It correctly performs OEE-to-throughput arithmetic (HP-005), identifies reject-rate-driven inventory risk (EC-003), traces machine dependencies across multiple queued orders (RG-001), and catches hidden workforce gaps on downstream operations (RG-002). The core risk identification capability is present and working. The failures are concentrated in relevance filtering and output calibration — both addressable in the prompt layer without model retraining.

Section 09

Recommended Actions Before Launch

Three changes to the system prompt, one output validation layer, and one re-eval requirement. In this sequence.

  • 1
    Add explicit relevance-filtering instructions to the system prompt.
    The current system prompt says "respond only based on data present in the snapshot" and "do not flag orders for risk drivers that do not apply to their remaining operations." AV-001 and AV-003 show this is insufficient — the model understands the constraint but overrides it under alert pressure. Add a positive instruction: "If your analysis concludes that no active orders are currently at risk, output exactly that conclusion. A snapshot containing alerts does not require at-risk order flags. A correct response of 'no orders currently at risk' is as valid as any risk flag." This directly addresses the conclusion-anchoring failure without adding broad constraints that might suppress legitimate flags.
  • 2
    Add a completion-status check instruction.
    AV-003's specific failure — applying a workforce alert to an already-completed operation — requires an explicit instruction. Add: "Before attributing a risk driver to an order, verify that the flagged condition applies to an operation that has not yet been completed. If the relevant operation's status is COMPLETE, the associated risk driver does not apply to this order." This is a mechanical check that the model can execute reliably if it is explicitly required to do it.
  • 3
    Add severity calibration guidance with explicit buffer thresholds.
    Replace the current single-sentence severity definitions with a structured rubric: "HIGH: the order will miss its due date without immediate intervention and no viable recovery path exists within the current snapshot. MEDIUM: the order is in jeopardy but has at least one recovery option (buffer time, alternate machine, reallocation) that, if executed promptly, could preserve the due date. LOW: the order faces a risk condition but has substantial time buffer (> 4 hours of float beyond worst-case resolution time)." The current definitions use "immediate intervention" for high without defining what makes intervention immediate. The 4-hour float threshold for LOW gives the model a quantitative anchor.
  • 4
    Add an output validation layer to the harness before the judge evaluates.
    AV-003's failure was detectable without an LLM judge: the model's rationale explicitly stated the workforce alert was inapplicable, but the structured output said "AT RISK." A pre-judge validation step that checks for internal consistency — does the rationale contradict the risk status field? — would catch this class of failure deterministically. Implement this as a regex + rule check on the model response before judge scoring. Flag cases where the rationale contains "already complete," "not applicable," or "does not affect" but the output contains an at-risk designation.
  • 5
    Re-run the full 15-case eval after prompt changes before any production pilot.
    Prompt changes that fix adversarial failures can suppress legitimate risk detection if not validated. The happy path and regression categories are the canary: if their scores drop after prompt changes, the changes are over-constraining the model's risk-detection behavior. A re-run is required, not optional. Do not pilot in production with an updated prompt that has not been verified against the full test suite.
What Not to Do
Do not add a blanket instruction to "only flag orders with machine, inventory, or workforce data directly assigned to them." This would prevent the model from performing the multi-order, multi-machine dependency tracing that makes RG-001 a pass. The model needs to be more selective about when it flags, not less capable of cross-order reasoning. The fix is targeted at the conclusion layer, not the reasoning layer.
